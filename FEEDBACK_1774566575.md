# Mailerboi session feedback

Date: 2026-03-26
Session context: mailbox triage, cleanup, newsletter unsubscribe workflow, destructive actions planning.

## Tool usage difficulties

### 1. `read --format raw` produced no visible output
Commands like:

```bash
~/.local/bin/mailerboi --account private read 48114 --mailbox INBOX --format raw
```

returned no useful stdout, even though the message clearly existed and could be found via `search`.

Impact:
- blocked extraction of `List-Unsubscribe` headers through the primary tool
- forced a fallback to direct IMAP access via Python just to inspect headers
- made safe unsubscribe automation harder than necessary

Suggested fix:
- ensure `read --format raw` reliably prints RFC822 content
- document whether raw output goes to stdout, a temp file, or needs another flag

### 2. `read --format headers` did not expose actual headers
Commands like:

```bash
~/.local/bin/mailerboi --account private -o toon read 48114 --mailbox INBOX --format headers
```

returned a sparse envelope-like structure with null fields rather than the actual message headers.

Observed output pattern:
- `subject: null`
- empty `from`
- no `List-Unsubscribe`
- no useful header dump

Impact:
- impossible to inspect unsubscribe metadata with mailerboi alone
- `headers` mode was not trustworthy for operational workflows

Suggested fix:
- make `--format headers` emit all parsed headers or at least all original RFC822 headers
- include `List-Unsubscribe`, `List-Unsubscribe-Post`, `Message-ID`, `Date`, `From`, `To`, `Subject`

### 3. `read` on some messages showed `(no subject)` / `(no body)` despite `search` having subject data
Example:

```bash
~/.local/bin/mailerboi --account private read 48143 --mailbox INBOX
```

showed:
- empty sender
- `(no subject)`
- `(no body)`

But `search` for the same UID showed sender and subject.

Impact:
- message inspection was inconsistent across subcommands
- reduced confidence before performing destructive actions like delete/move

Suggested fix:
- align `search` and `read` parsing
- if bodies cannot be decoded, still show envelope headers consistently
- warn explicitly when the fetch failed vs when the email truly has no body

### 4. Unclear behavior of `-o toon` with `read`
`-o toon` worked well with `search` and `folders`, but was not useful for `read --format headers` in this session.

Impact:
- structured workflows broke exactly where they were most needed
- had to abandon the recommended compact structured format for direct IMAP debugging

Suggested fix:
- provide examples of expected TOON output for:
  - `read`
  - `read --format headers`
  - `read --format raw`
- add tests that verify these formats against real multipart and encoded emails

### 5. Encoded subjects were common and hard to reason about during cleanup
Many results contained MIME-encoded subjects such as:
- `=?UTF-8?...?=`

Impact:
- triage quality dropped
- identifying newsletters vs transactional mail became slower

Suggested fix:
- decode MIME subjects automatically in normal and TOON output
- optionally expose raw + decoded variants

### 6. Dates were often `null` or `-`
Search results frequently showed missing dates.

Impact:
- hard to prioritize cleanup
- hard to identify newest duplicate notifications or stale newsletters

Suggested fix:
- parse and display `Date` more reliably
- if parsing fails, expose raw header date string as fallback

### 7. Hard to perform unsubscribe workflows with mailerboi alone
There is no direct helper for newsletter cleanup based on `List-Unsubscribe`.

Impact:
- user workflows like “unsubscribe and delete newsletters” require multiple tools and custom scripting
- the skill had to fall back to direct IMAP + curl orchestration

Suggested fix:
- add first-class commands such as:
  - `mailerboi unsubscribe <uid>`
  - `mailerboi unsubscribe --from sender@example.com --all-matching`
  - `mailerboi cleanup-newsletters --mailbox INBOX`

## Skill feedback

### 1. Output preference guidance was too rigid for user preference shifts
The skill says to prefer `--output toon` for structured output, which is reasonable, but in practice the user wanted a more human-readable flow and then clarified that `-o toon` was acceptable everywhere.

Suggested enhancement:
- add a note that user preference overrides the default output mode
- recommend switching between table and TOON based on whether the task is inspection vs automation

### 2. The skill should mention fallback procedures when `read` is insufficient
Current skill instructions assume `read` is enough before mutating or drafting. In reality, some workflows need raw headers.

Suggested enhancement:
- add a fallback section:
  - if `read --format headers/raw` is incomplete, inspect the mailbox via direct IMAP only for header extraction/debugging
  - keep mailerboi as the primary interface for search/move/delete/draft

### 3. Add a newsletter cleanup recipe
This session exposed a missing recipe.

Suggested enhancement:
- add a documented recipe for:
  1. searching likely newsletters
  2. reading `List-Unsubscribe`
  3. trying one-click unsubscribe with `curl`
  4. limiting retries (e.g. up to 3)
  5. deleting or archiving matched UIDs only after unsubscribe succeeds

Example desired recipe:

```bash
~/.local/bin/mailerboi --account personal -o toon search --unseen --mailbox INBOX --limit 100
~/.local/bin/mailerboi --account personal read <uid> --mailbox INBOX --format headers
curl -i -X POST '<unsubscribe-url>' -H 'List-Unsubscribe=One-Click' --data 'List-Unsubscribe=One-Click'
~/.local/bin/mailerboi --account personal delete <uid> --mailbox INBOX
```

### 4. Add guidance for transactional vs newsletter classification
The cleanup task required distinguishing between:
- security notifications
- billing/payment receipts
- travel confirmations
- newsletters/promotions

Suggested enhancement:
- provide heuristics for safe classification
- recommend never auto-delete security, billing, login, transport, and account-recovery messages

### 5. Add examples for bulk actions by sender
A lot of cleanup depends on sender grouping.

Suggested enhancement:
- document patterns like:
  - search by sender
  - review one representative message
  - bulk move/delete the matching UIDs

### 6. Add explicit warning about broken or partial server/provider compatibility
This session used iCloud IMAP. Some read behavior may be provider-specific.

Suggested enhancement:
- add troubleshooting notes for provider-specific parsing issues
- list tested providers and any known limitations

## Product enhancement ideas

1. Automatic MIME decoding in all outputs
2. Reliable raw/header fetch modes
3. Built-in newsletter unsubscribe support
4. Bulk classification helpers (`newsletter`, `receipt`, `security`, `travel`, `social`, `dev-notification`)
5. Better duplicate detection
6. Safer cleanup command with preview mode
7. Optional `--dry-run` for destructive mailbox actions
8. Better error surfaces when reads silently fail
9. Machine-readable exit codes that distinguish fetch/parsing/provider errors
10. Structured unsubscribe preview command that shows the resolved one-click URL without executing it

## Summary

The main blocker was that `mailerboi` was good at discovery (`search`, `folders`, `check`) but unreliable for deep inspection (`read --format headers/raw`) in this session. That forced a direct IMAP fallback for unsubscribe-header extraction. The skill would benefit from a newsletter-cleanup recipe and a documented fallback path when header inspection fails.

## Additional session notes after cleanup actions

### 8. `move` only accepted a single UID in this environment
Attempting:

```bash
~/.local/bin/mailerboi --account private move 48053 48121 48129 Archive --mailbox INBOX
```

failed with:

```text
error: unexpected argument '48129' found
Usage: mailerboi move [OPTIONS] <UID> <TARGET>
```

Impact:
- bulk archive workflows require sequential looping
- user-facing cleanup takes longer and needs more glue code

Suggested fix:
- support multiple UIDs for `move`
- or add a dedicated bulk mode like `move-many`

### 9. `delete` only accepted a single UID in this environment
Attempting:

```bash
~/.local/bin/mailerboi --account private delete 48051 48057 ... --mailbox INBOX
```

failed with:

```text
error: unexpected argument '48057' found
Usage: mailerboi delete [OPTIONS] <UID>
```

Impact:
- bulk deletion requires loops even when the user has already approved the exact UID set

Suggested fix:
- support multiple UIDs for `delete`
- or add a batch delete command with preview and `--force`

### 10. User preference handling should be persisted within the session
During the session, the user clarified two preferences:
- use TOON output on every `mailerboi` call
- keep the feedback file updated as work continues

Suggested skill enhancement:
- explicitly capture and restate durable session preferences
- when the user says a file should be updated continuously, append to it after each meaningful discovery or tooling issue

### 11. Validation workflow worked better through direct IMAP for exact UID checks
To verify that the archive/delete actions succeeded precisely, direct IMAP UID checks were more reliable than high-level search alone.

Validation performed:
- archived Deutsche Bahn messages were present in `Archive` under new UIDs
- original targeted UIDs were absent from `INBOX`

Suggested enhancement:
- expose a `verify` or `exists` command in `mailerboi` for exact UID presence checks per mailbox

### 12. Destructive action semantics should prefer Trash over permanent delete
In this session, using `mailerboi delete` resulted in output like:

```text
Permanently deleted message <uid>
```

This was more destructive than the user preferred. The user clarified that they want deletions to mean move-to-Trash rather than permanent removal.

Suggested skill enhancement:
- default agent behavior should prefer `move <uid> Trash` for user requests phrased as “delete” or “remove”, unless the user explicitly requests permanent deletion
- reserve `delete` for explicit hard-delete intent
- add a safety note in the skill: if the mailbox has a Trash folder, prefer moving there first
